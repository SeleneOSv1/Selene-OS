- self-authored Apple build instructions can omit explicit cleanup of repo-local `xcodebuild` artifact trees such as `apple/mac_desktop/build` or `apple/iphone/build`, which risks violating clean-tree law even when product edits are otherwise correct

## Agent-Law Change

H256 permanently refines repository agent law by requiring the following for self-authored build instructions:

- if the instruction runs `xcodebuild`, Codex must remove the relevant repo-local build artifact tree before the first clean-tree check
- if the instruction runs `xcodebuild`, Codex must remove the relevant repo-local build artifact tree again after each `xcodebuild` before final cleanliness verification
- for Apple target work, the default relevant cleanup targets are `apple/mac_desktop/build` and `apple/iphone/build` when the task touches those targets
- build artifact cleanup applies only to build-output trees and does not authorize deletion outside the task’s lawful build-output directories
- if the instruction cites preserved prior landed builds, Codex must include explicit same-task git verification commands for each cited prior landed build hash
- pinning current `HEAD` or `origin/main` is not sufficient prior-hash verification when earlier landed builds are also cited as preserved truth
- the required prior-hash verification must map each cited build identifier to its current repo hash through exact same-task git output such as exact `git log --format=%H --grep='^H253:' -n 1` or an exact ancestor query

The following are now permanently forbidden under repository agent law:

- allowing repo-local `xcodebuild` artifact trees to remain in place before start-of-run or end-of-run cleanliness checks
- citing preserved prior landed builds in a self-authored build instruction without explicit same-task git verification of each cited prior build hash
- treating `HEAD == ...` as sufficient proof for earlier cited landed builds when those earlier builds are named separately in the instruction
