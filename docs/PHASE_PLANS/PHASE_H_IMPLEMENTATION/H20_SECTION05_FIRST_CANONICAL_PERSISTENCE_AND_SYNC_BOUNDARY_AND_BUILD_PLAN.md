## Canonical Persistence Boundary
The first canonical persistence boundary is the adapter-owned deterministic persistence state machine rooted in `AdapterPersistenceState` in `crates/selene_adapter/src/lib.rs`. Inside this boundary, the lawful durable carriers are the `outbox_records` map, the append-only `operation_journal`, the `authoritative_outcomes` cache, the append-only `audit_trail`, and the explicit `recovery_mode` state. This boundary begins when a request is normalized to one `operation_id` plus one idempotency identity inside `prepare_persistence_operation`, and it ends when `finalize_persistence_operation` records one of the already-modeled deterministic outcomes: authoritative acknowledgement, retained retryable outbox state, terminal authoritative failure, or quarantine.

`try_reuse_persisted_authoritative_result` is inside the canonical persistence boundary because it reuses already-committed authoritative cloud outcomes using the same idempotency identity. It is not a second authority path. `reconcile_pending_outbox_records` is also inside the canonical persistence boundary because it replays or rejects existing pending state through the same persistence rules, the same recovery posture, and the same idempotency identity rather than inventing a second replay path. `load_persistence_state_from_disk` and `save_persistence_state_to_disk_locked` are also inside this boundary because restart recovery, corruption quarantine, and durable replacement are part of persistence correctness, not optional convenience behavior.

The canonical persistence boundary must preserve the current published repo truth that cloud-authoritative execution remains the only source of authoritative outcomes. Local persistence may assist retry, dedupe reuse, reconciliation, and recovery, but it may not elevate local state to authoritative truth. If local state is stale, mismatched, or corrupt, the boundary must reuse a previously committed authoritative result, request fresh state, quarantine, or refuse. It must not silently merge, guess, or continue by best effort.

The only runtime-facing contract that may leave this canonical persistence boundary is `PersistenceExecutionState` in `crates/selene_kernel_contracts/src/runtime_execution.rs`. That contract may carry persistence posture to downstream deterministic consumers, but it must not redefine persistence truth upstream. Audit implications are part of the same boundary: reconciliation decisions, dedupe outcomes, quarantine actions, acknowledgement outcomes, and recovery-mode transitions must remain reconstructable from the journal plus audit trail that the adapter persistence state machine already owns.

## Canonical Sync Boundary
The first canonical sync boundary is the typed phone-local artifact sync queue lifecycle exposed by `Ph1MobileArtifactSyncRepo` in `crates/selene_storage/src/repo.rs`. This boundary is non-authoritative with respect to cloud execution outcomes, but it is authoritative for local sync-work bookkeeping. Current published repo truth already gives this boundary deterministic lifecycle edges for dequeue with worker lease, acknowledgement by worker, fail with retry window, replay-due discovery, dead-letter discovery, and dead-letter commit.

The storage tests in `crates/selene_storage/tests/ph1_voice_id/db_wiring.rs` prove that this boundary already enforces replay only after lease expiry, worker-bound acknowledgement safety, retry-window persistence after failure, and durable queue state across multiple attempts. That matters because the first canonical sync boundary must be grounded in a real typed queue lifecycle, not in UI state or later generalized cluster coordination.

Sync observation stops at derived queue counters, replay-due or dead-letter classification, and downstream improvement observation such as `emit_sync_improvement_events` and `maybe_run_builder_for_sync_improvements` in `crates/selene_adapter/src/lib.rs`. Deterministic sync mutation begins at the storage repo commit edges: `mobile_artifact_sync_dequeue_batch_row`, `mobile_artifact_sync_ack_commit_row`, `mobile_artifact_sync_fail_commit_row`, and `mobile_artifact_sync_dead_letter_commit_row`. The builder is downstream of this canonical sync boundary. It may consume severe sync evidence, but it must not become the owner of retry, replay, or acknowledgement truth.

The persistence and sync boundaries meet at observation and projection, not shared authority. Queue health and queue backlog may influence downstream reporting and improvement signals, and persistence posture may be projected through `PersistenceExecutionState`, but neither boundary may turn the other into a parallel authority path.

## Deterministic Invariants
Current published repo truth must preserve one cloud-authoritative result for one logical idempotency identity. Outbox replay must reuse the same `operation_id` and the same idempotency identity, not invent a new logical action. Duplicate execution must reuse a prior authoritative outcome or refuse, never produce an extra mutation. Reconciliation must remain auditable through append-only journal and audit entries. Recovery-mode transitions must remain explicit and replayable. Stale, mismatched, or corrupt local state must quarantine or refuse rather than silently merge.

Sync queue acknowledgement must remain worker-safe and lease-aware. Stable ordering must remain bound to session, turn, device, and device-turn sequence inputs. Downstream runtime consumers may observe persistence posture through `PersistenceExecutionState`, but they must not mutate persistence truth by bypass path. The system must fail closed on corruption, stale replay, worker mismatch, or uncertain authority. Transcript cleanup triggered by reuse must remain downstream of authoritative reuse rather than a second execution lane. Any cross-node dedupe handling must preserve the already-exposed `cross_node_dedupe_applied` contract meaning instead of inventing a second shared contract.
