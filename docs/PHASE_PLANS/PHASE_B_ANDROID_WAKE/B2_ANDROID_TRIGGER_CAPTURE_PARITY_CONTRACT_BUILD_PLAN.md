D) PLATFORM-LAW CONTRACT REQUIREMENTS
- Android 12+ background-start restrictions
- B2 must encode that background context alone is not enough to start microphone capture or microphone FGS.
- `AndroidClientVisibilityState=BACKGROUND` with `trigger_class=WAKE_PHRASE` must not silently imply lawful wake capture.
- Android 14+ microphone foreground-service type/permission requirements
- B2 must require `service_type_microphone_declared=true`, `fgs_permission_granted=true`, and lawful start context before any `FGS_MICROPHONE_ELIGIBLE` result.
- Android 14+/15 boot-start microphone restrictions
- `trigger_class=BOOT_RESTORE` may support config restore, readiness recomputation, and boot-restoration receipts, but must not yield microphone capture start legality by itself.
- While-in-use permission realities
- `GRANTED_WHILE_IN_USE` is not equivalent to unlimited background microphone legality; B2 must keep permission state separate from start legality.
- Privacy-toggle / silent-audio posture
- Privacy toggle off and silent-audio delivery are separate typed states. They must not be collapsed into generic failure.
- Audio-input sharing / contention posture
- Contention is a first-class operational state and must produce explicit posture/receipt/fallback outputs.
- Wake-lock boundedness posture
- B2 must encode wake lock as an operational boundedness contract only. Wake lock may never imply trust, permission, or legal capture authority.

F) PARITY / FALLBACK CONTRACT
- `desktop-full-parity`
  - means always-listening or wake behavior is lawful and operationally supported to the same class as desktop/reference path
  - not an acceptable Android assumption by default
- `android-lawful-parity`
  - means Android delivers wake/capture behavior matching desktop intent only where Android platform law actually allows it
  - acceptable later release target
- `android-explicit-only-fallback`
  - means Android cannot lawfully provide always-listening/wake in the current posture and degrades to visible explicit capture only
  - acceptable later release target
- `android-blocked-platform-posture`
  - means current platform posture or policy forbids lawful operation
  - acceptable later release target
- Acceptable later B3/B4 targets:
- `ANDROID_LAWFUL_PARITY`
- `ANDROID_EXPLICIT_ONLY_FALLBACK`
- `ANDROID_BLOCKED_PLATFORM_POSTURE`
- `DESKTOP_FULL_PARITY` is not a required Android target if platform law forbids it.
