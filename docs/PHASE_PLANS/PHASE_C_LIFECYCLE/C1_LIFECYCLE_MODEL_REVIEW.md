C) CANONICAL C1 LIFECYCLE DESIGN
1. artifact creation
- canonical state: `CREATED`
- meaning: artifact exists as a candidate object with identity, lineage root, and provenance, but is not yet published or distributed
2. publication
- canonical state: `PUBLISHED`
- meaning: artifact is admitted into the authoritative distribution domain and becomes eligible for downstream distribution rules
3. distribution
- canonical state: `DISTRIBUTED`
- meaning: artifact has been made available to an operational target or channel, but is not yet installed or applied
4. installation
- canonical state: `INSTALLED`
- meaning: artifact payload or representation is present at the target environment/device/storage location but is not yet applied
5. apply request
- canonical state: `APPLY_REQUESTED`
- meaning: an operational request exists to move an installed artifact into an active or locally applied posture
6. apply receipt
- canonical state: `APPLY_CONFIRMED`
- meaning: the apply operation has emitted its operational receipt and local confirmation
7. activation
- canonical state: `ACTIVE`
- meaning: artifact is canonically active and may be used at runtime
8. runtime use
- canonical state: `IN_RUNTIME_USE`
- meaning: an active artifact is actually being consumed in runtime execution
9. replacement
- canonical state: `REPLACED`
- meaning: artifact is no longer the active choice because a successor artifact replaced it under canonical lineage rules
10. rollback
- canonical state: `ROLLED_BACK`
- meaning: artifact or transition was reversed in favor of a previously valid target under explicit rollback rules
11. revocation
- canonical state: `REVOKED`
- meaning: artifact is no longer lawful or admissible for future use regardless of install or archive status, unless a governed historical verification path exists for proof-only visibility
12. expiry
- canonical state: `EXPIRED`
- meaning: artifact passed its allowed use window and may not continue active/runtime use
13. archival
- canonical state: `ARCHIVED`
- meaning: artifact is no longer active but is intentionally retained in cold/archive posture with restorability metadata
14. purge/delete
- canonical state: `PURGED`
- meaning: artifact or lifecycle subject has been terminally removed under explicit purge/delete semantics
15. restoration/recovery
- canonical state: `RESTORED`
- meaning: an archived or previously inactive lifecycle subject has been restored into a lawful non-terminal posture, subject to trust/governance/proof rules
- transition concepts that must exist across these states:
- lifecycle transitions must be explicit, not inferred
- replacement, rollback, revocation, expiry, archive, purge, and restore must preserve lineage and evidence
- restoration must never bypass existing Phase A trust/proof/governance/law boundaries
- memory retention lifecycle must align to these concepts but remain a distinct owned domain

D) LIFECYCLE OWNERSHIP MODEL
- local/client operational lifecycle
  - owns distribution-to-install-to-apply-request-to-apply-confirmed operational truth, including local receipts and local restoration posture where applicable
  - does not own trust truth, proof truth, governance truth, or law truth
- storage lifecycle
  - owns canonical persistence of lifecycle state, archival state, purge/delete truth, restorability metadata, and materialized history across tables
- trust/proof lifecycle
  - owns visibility and proof-linkage for lifecycle-significant transitions that matter to safety, replay, or historical correctness
  - must stay inside frozen Phase A trust/proof boundaries
- governance lifecycle
  - owns governed visibility and approval posture for activation, replacement, rollback, revocation, and recovery transitions where governance is required
- law visibility
  - owns final runtime posture visibility of lifecycle-derived risk where lifecycle state affects runtime safety
  - does not own lifecycle mechanics or lifecycle storage truth
- memory retention lifecycle
  - owns retention, decay, forget/delete, and restoration posture for memory-domain objects
  - must align with canonical lifecycle semantics for expiry/archive/purge/restore, but must not erase the distinction between memory and artifact domains
