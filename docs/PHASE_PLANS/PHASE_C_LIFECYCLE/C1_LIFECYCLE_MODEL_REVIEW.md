B) CURRENT LIFECYCLE STATE
**WORLD-CLASS UPGRADE — Multi-Plane Lifecycle Architecture**
- C1 splits lifecycle into four related but non-identical planes:
- `ARTIFACT_IDENTITY_LIFECYCLE`
- `TARGET_DEPLOYMENT_LIFECYCLE`
- `RUNTIME_USE_LIFECYCLE`
- `MEMORY_LIFECYCLE`
- These planes must not be collapsed into one flat state machine.
- One artifact may be `ACTIVE` at artifact-global scope while still only `INSTALLED` or `APPLY_REQUESTED` on some targets.
- Runtime use may be active on one target while another target remains only distributed or archived.
- Memory-domain objects must align to lifecycle discipline without being mistaken for artifact identity lifecycle subjects.

**WORLD-CLASS UPGRADE — Lifecycle Scope Model**
- C1 defines explicit lifecycle scope classes:
- `ARTIFACT_GLOBAL_SCOPE`
- `TARGET_DEPLOYMENT_SCOPE`
- `DEVICE_LOCAL_SCOPE`
- `SESSION_SCOPE`
- `TENANT_SCOPE`
- `ENVIRONMENT_SCOPE`
- `CLUSTER_SCOPE`
- `PUBLISHED`, `REVOKED`, and `EXPIRED` may exist at `ARTIFACT_GLOBAL_SCOPE`, `TENANT_SCOPE`, `ENVIRONMENT_SCOPE`, or `CLUSTER_SCOPE` depending on subject class and governing authority.
- `DISTRIBUTED`, `INSTALLED`, `APPLY_REQUESTED`, and `APPLY_CONFIRMED` are primarily `TARGET_DEPLOYMENT_SCOPE` or `DEVICE_LOCAL_SCOPE`.
- `IN_RUNTIME_USE` may exist at `SESSION_SCOPE`, `DEVICE_LOCAL_SCOPE`, `TENANT_SCOPE`, or `ENVIRONMENT_SCOPE` depending on the runtime substrate.
- `ARCHIVED`, `PURGED`, and `RESTORED` may exist at artifact-global, tenant, environment, or device-local scope, but must never silently cross scopes without an explicit transition.
- Scope widening and scope narrowing are lifecycle-significant changes and must not be inferred.

**WORLD-CLASS UPGRADE — State Taxonomy by Terminality / Restorability / Proof Visibility**
| State | Terminal or non-terminal | Restorable or non-restorable | Proof-visible or operational-only | Globally authoritative or target-local operational |
| --- | --- | --- | --- | --- |
| `CREATED` | non-terminal | not directly restorable; it is a live pre-publication state | operational-only unless explicitly governed | globally authoritative for identity existence only |
| `PUBLISHED` | non-terminal | not a restoration target; it is a live authoritative state | proof-visible when publication is safety, trust, or governance relevant | globally authoritative |
| `DISTRIBUTED` | non-terminal | restorable only as a deployment-instance state, not as artifact-global truth | operational-only by default | target-local operational |
| `INSTALLED` | non-terminal | restorable only as a deployment-instance state | operational-only by default | target-local operational |
| `APPLY_REQUESTED` | non-terminal | restorable only as an operational request state | operational-only by default | target-local operational |
| `APPLY_CONFIRMED` | non-terminal | restorable only as a target deployment lifecycle subject | operational-only unless a governed apply confirmation is safety-relevant | target-local operational |
| `ACTIVE` | non-terminal | may later become archived or replaced, but `ACTIVE` itself is not a restoration target | proof-visible when activation matters to runtime safety or compliance | globally authoritative or target-authoritative depending subject scope |
| `IN_RUNTIME_USE` | non-terminal | not a restoration target; it is ephemeral runtime truth | operational-only unless runtime use transition is safety-significant | target-local or session-local operational |
| `REPLACED` | non-terminal historical state | not directly restorable; restoration must target a lawful subject state such as `ARCHIVED` or `RESTORED` | proof-visible | globally authoritative or target-authoritative historical state |
| `ROLLED_BACK` | non-terminal historical state | not directly restorable; later restore must follow explicit rules | proof-visible | globally authoritative or target-authoritative historical state |
| `REVOKED` | non-terminal historical state for visibility, terminal for future activation | non-restorable into `ACTIVE` without explicit governed replacement or re-approval path | proof-visible | globally authoritative or scoped authoritative depending revocation scope |
| `EXPIRED` | non-terminal historical state for visibility, terminal for future active use | non-restorable into `ACTIVE` unless a fresh governed lifecycle subject is admitted | proof-visible where safety/compliance relevant | globally authoritative or scoped authoritative depending expiry scope |
| `ARCHIVED` | non-terminal | restorable | proof-visible when compliance, safety, or restoration lineage matters | globally authoritative or scoped authoritative archival posture |
| `PURGED` | terminal | non-restorable as the same lifecycle subject | proof-visible where governed, compliance, or legal-hold semantics require it | authoritative terminal state |
| `RESTORED` | non-terminal | itself may later archive, expire, or be replaced again | proof-visible when restoration is safety/compliance/governance relevant | globally authoritative or scoped authoritative depending restored scope |

**WORLD-CLASS UPGRADE — Lifecycle Transition Guard Categories**
- `STORAGE_GUARD`
  - required when persistence, archival, purge/delete, or restorability metadata is affected
- `TRUST_GUARD`
  - required when lifecycle transition changes admissibility, trust posture, or authenticity expectations
- `PROOF_GUARD`
  - required when transition must become canonical lifecycle evidence visible through Phase A proof discipline
- `GOVERNANCE_GUARD`
  - required when activation, replacement, rollback, revocation, expiry, or restore requires governed authorization or visibility
- `LAW_VISIBILITY_GUARD`
  - required when lifecycle transition must be visible to runtime law posture
- `RETENTION_GUARD`
  - required when retention class, hold state, archive eligibility, or delete eligibility changes
- `RESTORATION_GUARD`
  - required when restoring from archive, cold storage, or historical visibility back toward usable posture

- Receipt presence is not a substitute for authoritative lifecycle state.
- Missing receipt is operationally relevant, but receipt alone may not create an authoritative lifecycle transition.

**WORLD-CLASS UPGRADE — Multi-Target Deployment Model**
- C1 distinguishes:
- `artifact_global_lifecycle_state`
- `per_target_deployment_instance_lifecycle_state`
- `runtime_consumption_state`
- One artifact may be globally `PUBLISHED` or `ACTIVE` while target A is only `INSTALLED`, target B is `APPLY_REQUESTED`, and target C is already `IN_RUNTIME_USE`.
- Deployment instance lifecycle must never overwrite artifact-global lifecycle truth.
- Runtime consumption state must never be mistaken for publication, activation, or revocation truth.

1. artifact creation
- canonical state: `CREATED`
- meaning: artifact exists as a candidate object with identity, lineage root, and provenance, but is not yet published or distributed
2. publication
- canonical state: `PUBLISHED`
- meaning: artifact is admitted into the authoritative distribution domain and becomes eligible for downstream distribution rules
3. distribution
- canonical state: `DISTRIBUTED`
- meaning: artifact has been made available to an operational target or channel, but is not yet installed or applied
4. installation
- canonical state: `INSTALLED`
- meaning: artifact payload or representation is present at the target environment/device/storage location but is not yet applied
5. apply request
- canonical state: `APPLY_REQUESTED`
- meaning: an operational request exists to move an installed artifact into an active or locally applied posture
6. apply receipt
- canonical state: `APPLY_CONFIRMED`
- meaning: the apply operation has emitted its operational receipt and local confirmation
7. activation
- canonical state: `ACTIVE`
- meaning: artifact is canonically active and may be used at runtime
8. runtime use
- canonical state: `IN_RUNTIME_USE`
- meaning: an active artifact is actually being consumed in runtime execution
9. replacement
- canonical state: `REPLACED`
- meaning: artifact is no longer the active choice because a successor artifact replaced it under canonical lineage rules
10. rollback
- canonical state: `ROLLED_BACK`
- meaning: artifact or transition was reversed in favor of a previously valid target under explicit rollback rules
11. revocation
- canonical state: `REVOKED`
- meaning: artifact is no longer lawful or admissible for future use regardless of install or archive status, unless a governed historical verification path exists for proof-only visibility
12. expiry
- canonical state: `EXPIRED`
- meaning: artifact passed its allowed use window and may not continue active/runtime use
13. archival
- canonical state: `ARCHIVED`
- meaning: artifact is no longer active but is intentionally retained in cold/archive posture with restorability metadata
14. purge/delete
- canonical state: `PURGED`
- meaning: artifact or lifecycle subject has been terminally removed under explicit purge/delete semantics
15. restoration/recovery
- canonical state: `RESTORED`
- meaning: an archived or previously inactive lifecycle subject has been restored into a lawful non-terminal posture, subject to trust/governance/proof rules
- transition concepts that must exist across these states:
- lifecycle transitions must be explicit, not inferred
- replacement, rollback, revocation, expiry, archive, purge, and restore must preserve lineage and evidence
- restoration must never bypass existing Phase A trust/proof/governance/law boundaries
- memory retention lifecycle must align to these concepts but remain a distinct owned domain

**WORLD-CLASS UPGRADE — Replacement / Rollback Eligibility Matrix**
| Eligibility dimension | Replacement lawful? | Rollback lawful? | Notes |
| --- | --- | --- | --- |
| same artifact class | required | required | class drift is not lawful by default |
| same lineage family | required | required | lineage mismatch is a blocker |
| same scope | required | required | scope mismatch requires explicit governed transition, not silent replacement |
| same compatibility family | required | required | incompatible targets may not be replaced or rolled back silently |
| not revoked unless proof/history-only | required for active replacement | required for active rollback target | revoked subjects may remain historical only |
| not expired unless governed historical path only | required for active replacement | required for active rollback target | expired subjects may remain historical only |

**WORLD-CLASS UPGRADE — Revocation / Expiry / Archive Interaction Law**
- Revoked artifacts may remain proof-visible or archive-visible.
- Revoked artifacts may not silently become restorable into `ACTIVE` without governed replacement or re-approval rules.
- Expired artifacts may remain historical but not active.
- `ARCHIVED` is not the same as `REVOKED` or `EXPIRED`.
- Archive preserves restorability posture; revocation and expiry constrain or forbid active use.

**WORLD-CLASS UPGRADE — Purge / Delete Terminality Rule**
- `PURGED` is terminal.
- `PURGED` subjects may not be restored as the same lifecycle subject.
- Any later reappearance requires a new identity or an explicit governed reconstruction rule.
- Purge/delete must never masquerade as archive.

**WORLD-CLASS UPGRADE — Retention Class Model**
- Canonical lifecycle-aware retention classes are:
- `HOT_RETENTION`
- `WARM_RETENTION`
- `COLD_RETENTION`
- `ARCHIVE_RETENTION`
- `LEGAL_HOLD_RETENTION`
- `DELETE_ELIGIBLE`
- `CREATED`, `PUBLISHED`, `ACTIVE`, and `IN_RUNTIME_USE` usually imply `HOT_RETENTION` or `WARM_RETENTION` depending runtime criticality.
- `REPLACED`, `ROLLED_BACK`, `EXPIRED`, and `REVOKED` often transition toward `WARM_RETENTION`, `COLD_RETENTION`, or `ARCHIVE_RETENTION`.
- `ARCHIVED` maps naturally to `ARCHIVE_RETENTION`.
- `PURGED` is beyond retention and is terminal.
- Legal hold may override normal retention decay.

**WORLD-CLASS UPGRADE — Legal Hold / Preservation Hold Rule**
- C1 recognizes:
- `LEGAL_HOLD`
- `PRESERVATION_HOLD`
- Either hold may prevent purge/delete even when retention would otherwise expire.
- Holds affect lifecycle transition eligibility but do not create a new trust, proof, or enforcement path.

**WORLD-CLASS UPGRADE — Restoration Eligibility Model**
- States that may be restored:
- `ARCHIVED`
- certain non-terminal historical states where restoration is explicitly permitted by lifecycle policy
- States that may never be restored directly as the same subject:
- `PURGED`
- `REVOKED` into `ACTIVE` without governed replacement or re-approval
- `EXPIRED` into `ACTIVE` without a fresh governed lifecycle path
- Restoration requires fresh trust, governance, proof, retention, and storage guard checks where applicable.
- Restoration may not bypass frozen Phase A controls.

**WORLD-CLASS UPGRADE — Concurrency / Idempotency / Ordering Law**
- Repeated apply requests must be idempotent.
- Duplicate distribution/install/apply receipts must not create duplicate authoritative transitions.
- Concurrent replacement and rollback races must resolve under stable ordering or explicit conflict posture.
- Lifecycle transition handling must be idempotent and replay-safe.
- Ordering guarantees must be explicit wherever two transitions could conflict on the same lifecycle subject.

**WORLD-CLASS UPGRADE — Partial-Failure / Compensation Model**
- C1 explicitly recognizes partial failure cases:
- install succeeded but apply failed
- apply succeeded locally but activation was not admitted
- archive requested but proof or governance was incomplete
- rollback initiated but replacement already advanced
- These are lifecycle-visible compensation states and must not be hidden as local-only cleanup.
- Compensation may not invent new lifecycle truth; it must consume the canonical states and guards.

D) LIFECYCLE OWNERSHIP MODEL
**WORLD-CLASS UPGRADE — Cross-Subsystem Lifecycle Mapping**
- Wake artifacts consume the canonical lifecycle model through target deployment, apply, activation, rollback, archive, and restore semantics.
- Builder outputs consume the canonical lifecycle model through publication, replacement, rollback, revocation, and archival semantics.
- Learn artifacts consume the canonical lifecycle model through candidate creation, publication, replacement, rollback, and historical visibility semantics.

- Memory-domain objects consume the canonical lifecycle model only through aligned retention, expiry, archive, purge/delete, and restore semantics, while preserving a distinct memory ownership boundary.

- local/client operational lifecycle
  - owns distribution-to-install-to-apply-request-to-apply-confirmed operational truth, including local receipts and local restoration posture where applicable
  - does not own trust truth, proof truth, governance truth, or law truth
- storage lifecycle
  - owns canonical persistence of lifecycle state, archival state, purge/delete truth, restorability metadata, and materialized history across tables
- trust/proof lifecycle
  - owns visibility and proof-linkage for lifecycle-significant transitions that matter to safety, replay, or historical correctness
  - must stay inside frozen Phase A trust/proof boundaries
- governance lifecycle
  - owns governed visibility and approval posture for activation, replacement, rollback, revocation, and recovery transitions where governance is required
- law visibility
  - owns final runtime posture visibility of lifecycle-derived risk where lifecycle state affects runtime safety
  - does not own lifecycle mechanics or lifecycle storage truth
- memory retention lifecycle
  - owns retention, decay, forget/delete, and restoration posture for memory-domain objects
  - must align with canonical lifecycle semantics for expiry/archive/purge/restore, but must not erase the distinction between memory and artifact domains
