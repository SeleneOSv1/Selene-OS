C) A2 CANONICAL CONTRACT DESIGN
Contract separation note: the approval baseline must treat `ArtifactIdentity`, `ArtifactTrustBinding`, `ArtifactLifecycleState`, `ArtifactScope`, and `ArtifactLineage` as distinct contract concerns. Identity must not silently absorb trust-binding, certification, or lifecycle semantics if those concerns evolve independently. A compatibility wrapper may exist, but A2 must not require a permanent collapse into one struct.

Digest authority note: `ArtifactDigest` is the one canonical digest contract for A2. Any `artifact_hash` wording that survives in transitional wrappers or compatibility adapters is legacy alias language only and must not remain a second authoritative truth. `VerificationCacheBasis`, proof entries, and decision records must bind to `ArtifactDigest`, not to a loose hash string.

Identity/trust separation law: `ArtifactIdentity` must not remain the permanent home for signer identity, trust-root identity, certification state, or other trust-binding fields. `ArtifactTrustBinding` is the canonical home for trust attachment. If a combined view is needed during migration, it must be explicitly transitional and non-final.

D) CONTRACT OWNERSHIP MODEL
- shared canonical contract layer: all A2 type definitions are shared canonical contracts in the contract layer. Section 04 does not “own the type definitions themselves.” No subsystem may fork these definitions locally.
- Section 04 Authority: sole canonical creator of first-time authoritative `ArtifactVerificationResult`, `NegativeArtifactVerificationResult`, `VerificationCacheBasis`, `ArtifactTrustDecisionRecord`, authoritative selection of `TrustPolicySnapshot`, binding to `TrustSetSnapshot`, and publication of `artifact_trust_state` into the envelope.
- Section 03 ingress: may carry artifact references, claims, receipts, and scope inputs only; it must not create authoritative trust bindings, trust snapshots, or verification decisions.
- PH1.OS: may carry device/platform/receipt/attestation/compatibility posture only; it must not own or mint artifact trust contracts or authoritative verification decisions.
- PH1.J: serializes `ArtifactTrustProofEntry` and proof linkage for already-made authority decisions; it does not perform verification.
- PH1.GOV: consumes `ArtifactTrustDecisionRecord` directly or thin wrappers around the same canonical record for activation/deprecation/rollback decisions only.
- PH1.LAW: consumes `ArtifactTrustDecisionRecord` directly or thin wrappers around the same canonical record for final posture only; it does not redefine artifact verification.
- storage: owns durable persistence contracts for artifact ledger rows, digests, lifecycle states, trust roots, signer identities, trust-policy snapshots, trust-set snapshots, revocation events, cache rows, negative verification rows when policy permits, decision records, proof entries, and proof rows.
- executors/consumers: read-only consumers of `artifact_trust_state` and `ArtifactTrustDecisionRecord`; no first-time authoritative verification and no local schema forks.

G) DRIFT / DUPLICATION RISKS
- parallel verifier contracts: if PH1.GOV, device sync, builder, or simulation code get their own trust structs, A2 fails.
- duplicate artifact schemas: if learn/self-heal/simcat/builder keep inventing local identity fields instead of embedding `ArtifactIdentity`, drift is immediate.
- identity/trust/lifecycle collapse drift: if one combined struct keeps absorbing trust-binding and lifecycle semantics, contract meaning will rot and consumers will diverge.
- digest-boundary drift: if `artifact_hash` remains an overloaded catch-all instead of a typed digest model, manifest and payload verification will disagree across subsystems.
- legacy-hash alias drift: if `artifact_hash` is allowed to survive as a second authoritative field instead of a transitional alias only, cache, proof, and replay will fork.
- bundle/manifest drift: if bundle trust is allowed to bind manifest-only in one subsystem and payload-only in another without one shared contract, replay will stop matching execution truth.
- dependency drift: if compatibility and dependency posture stay implicit, artifacts can verify locally while still being invalid in governed runtime context.
- authority-domain collapse: if signing, certification, revocation, and verification authority domains are not represented separately, root-of-trust concentration will become invisible.
- trust-set snapshot drift: if version labels survive without one governed membership snapshot contract, cluster replay and historical verification will diverge.
- decision-record drift: if PH1.J, GOV, LAW, or executor consumers reconstruct artifact trust from partial local fields instead of one shared `ArtifactTrustDecisionRecord`, A3 will immediately fork semantics.
- proof-entry drift: if proof correlation depends on flat vectors or positional coupling instead of structured per-artifact entries, multi-artifact turns will become non-deterministic.
- PH1.OS trust drift: if receipt or attestation posture is allowed to masquerade as artifact verification, authority law is broken.
- adapter/client trust drift: any client-visible signer/root/certification decision type is a design bug.
- PH1.J schema drift: proof cannot invent its own artifact-trust vocabulary separate from A2.
- negative-result drift: if failed verification is treated as ephemeral in one path and durable in another, cache invalidation and replay semantics will fork.
- verification-basis drift: if different subsystems compute different basis fingerprints for the same decision, proof correlation and dedupe become non-deterministic.
- scope-fingerprint drift: if scope fingerprints are carried as loose strings instead of one typed contract, proof, cache, and replay will disagree on scope identity.
- contract freeze drift: if fields or enums change after approval without explicit review, A3/A4/A5 will wire against moving targets.
- storage drift: if `artifacts_ledger`, trust roots, revocations, and cache rows do not share the same IDs and version fields, replay breaks.
- builder/learn/selfheal special-case drift: current artifact-like types already differ; A2 must normalize them or A3 will multiply exceptions.
