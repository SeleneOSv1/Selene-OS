0. Master Law

PH1.D is not the brain.

PH1.D is not the router.

PH1.D is not the executor.

PH1.D is the governed LLM proposal gateway.

The master law is:

OpenAI / GPT-5.5 proposes.
PH1.D governs the provider call and validates provider shape.
PH1.N structures messy meaning candidates where relevant.
PH1.X validates the Request Decision Lattice.
Stack owners validate their own truth and gates.
PH1.WRITE produces final human output.
Authority + SimulationExecutor control protected execution.

Provider output is powerful.

Provider output is never authority.

1. Why The Old Engine D Must Be Modernized

The old Engine D concept was useful because it enforced:

schema-valid LLM output
one controlled boundary
fail-closed malformed output
non-authoritative provider behavior
bounded context
reason codes
logging
clarification discipline

Those rules remain valuable.

But the old concept is too narrow for modern Selene because it assumed:

one LLM router output per turn
four output types only
old pipeline ordering
LLM router as a major routing boundary
limited awareness of PH1.N, PH1.X lattice, PH1.WRITE, Celine, Quick Assist, search, memory, voice, and protected lanes

Modern Selene needs PH1.D to be provider infrastructure, not the request brain.

The modern replacement is:

PH1.D = Provider-Governed LLM Proposal Gateway

It supports many proposal types, but final validation always belongs to Selene canonical owners.

5. Correct Position In The Modern Pipeline

Modern flow:

User input
→ PH1.C / PH1.LANG / PH1.N prepare transcript, language, and meaning candidates where applicable
→ PH1.D invokes GPT-5.5 / approved provider through Provider Governance where useful
→ provider returns structured proposal
→ PH1.D validates provider shape and evidence envelope
→ PH1.X validates Request Decision Lattice
→ stack owner handles the task:
   PH1.E search/tool/file/source evidence
   PH1.M memory
   Access/Governance private data scope
   Authority + Simulation protected execution
   PH1.WRITE final output
→ Adapter transports
→ Desktop/iPhone render/speak only
→ audit records evidence

PH1.D may be used by many stacks.

PH1.D never replaces those stacks.

8. Schema Discipline

PH1.D must use schema-bound outputs wherever possible.

Rules:

structured outputs preferred over free text for internal proposals
unknown fields rejected unless schema allows them
missing required fields rejected
unknown enum values rejected
malformed output rejected
provider refusal handled explicitly
provider timeout handled explicitly
provider hallucinated authority rejected
provider tool execution attempt rejected unless tool proposal schema allows only proposal

Provider output that fails schema validation must not reach PH1.X, PH1.E, PH1.M, PH1.WRITE, or protected owners as trusted proposal.

11. Tool / Function Calling Rule

OpenAI tool/function calling may be used as a proposal interface.

It may propose:

tool name
arguments
missing fields
clarification need
reason summary

It must not execute protected work directly.

Correct pattern:

Model proposes function/tool arguments.
PH1.D validates schema.
PH1.E validates tool scope and parameters.
Access/Authority validates permission where needed.
SimulationExecutor executes protected action only if lawful.

Never:

Model function call → protected execution

That path is forbidden.

12. Streaming / Realtime Rule

PH1.D may support streaming and realtime providers.

But streaming output is not final output.

Rules:

partial provider text is not final answer
realtime transcript deltas are not committed turns
realtime VAD is evidence only
provider interruption events are evidence only
PH1.WRITE owns final answer
PH1.K/PH1.C/PH1.X own voice control/meaning validation

A ProviderStreamFinalizationPacket or equivalent must exist before final user-visible output is accepted where streaming is used.

13. Fallback and Model Policy

Rules:
